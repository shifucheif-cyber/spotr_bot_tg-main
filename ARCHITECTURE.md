
# Описание архитектуры и логики проекта spotr_bot_tg-main

## 1. Общая цель

Бот — это мультидисциплинарный спортивный аналитик для Telegram, который:

- Принимает от пользователя запрос (дисциплина, команды, дата)

## 2. Основные модули и их роли

- Главный входной файл, FSM для Telegram-бота

- Логика поиска и валидации источников

- Приоритизация по региону, fallback на альтернативный регион

### services/data_fetcher.py

- Кэширование результатов анализа

- Формирует payload для LLM (отчёты, тотал, H2H, fallback)

### services/betting_calculator.py

- Извлечение вероятности, тотала, счёта, % ставки из ответа LLM

### services/football_service.py, hockey_service.py и др.

- Специализированные fetcher'ы для каждой дисциплины

- Используют data_fetcher и search_engine

### data_router.py

- Роутинг: определяет, какой fetcher использовать для дисциплины

- Асинхронный вызов нужного метода

### services/match_finder.py

- Валидация события, нормализация названий, разбор участников

### services/user_store.py


1. Пользователь отправляет запрос (дисциплина, команды, дата)
2. FSM в bot.py собирает все параметры
3. Вызов validate_match_request (search_engine) — валидация события, определение региона
4. Вызов get_match_data (data_router) — выбор fetcher'а по дисциплине
5. Fetcher (например, football_service) вызывает data_fetcher.fetch_match_analysis_data
6. data_fetcher собирает данные из search_engine, профильных библиотек, кэширует результат
7. Формируется payload для LLM (все источники, fallback, тотал, H2H)
8. Вызов LLM (Google Gemini ≥2.0, fallback Groq/SambaNova/DeepSeek)
9. Ответ LLM парсится через betting_calculator (извлекаются все ключевые поля)
10. bot.py формирует финальный ответ и отправляет пользователю

- Нет зацикливания: каждый этап вызывается только один раз, поиск прекращается при достижении min_sources

- Все даты приводятся к московскому времени

- Все переменные окружения вынесены в .env

- Кэширование анализа матчей (TTL=2 дня)

- Строгий формат вывода для пользователя

## 5. Как поддерживать описание

## 7. Детализированный разбор этапов pipeline и используемых инструментов

## 7.1. Этапы обработки запроса

### 1. Получение запроса пользователя

- Пользователь вводит дисциплину, команды/игроков, дату события через Telegram-бота.

- FSM (Finite State Machine) в `bot.py` поэтапно собирает все параметры.

### 2. Валидация события и участников

- Вызов `validate_match_request` из `services/search_engine.py`:

  - Проверяет, существует ли событие и корректны ли команды.

  - Определяет регион (RU/International) по ключевым словам, участникам, дисциплине.

  - Использует функции нормализации из `services/match_finder.py`.

### 3. Поиск и сбор данных

- Вызов `get_match_data` из `data_router.py`:

  - Определяет нужный fetcher по дисциплине (например, football_service, cs2_service и т.д.).

  - Fetcher вызывает `fetch_match_analysis_data` из `services/data_fetcher.py`.

#### 3.1. Поисковые этапы (services/search_engine.py)

- **Порядок поиска:**


  1. **Профильные библиотеки** (если есть):

  - Прямой парсинг/запрос к специализированным библиотекам (например, sportsipy, hltv, TheSportsDB, cloudscraper, BeautifulSoup).


  2. **DuckDuckGo (DDG)**:

  - Используется через библиотеки `ddgs` или `duckduckgo_search`.

  - Поиск по ключевым словам, дисциплине, участникам, дате.

  - Результаты валидируются по доменам и содержимому.


  3. **Serper**:

  - Официальный API поиска Google (`serper.dev`).

  - Используется для расширенного поиска, если профильные библиотеки и DDG не дали достаточно данных.


  4. **Exa/Tavily**:

  - Используются как резервные источники для сбора аналитических снитпетов и расширенной информации.


- **Fallback:**

  - Если после всех этапов не найдено достаточно валидированных источников, берутся лучшие совпадения из невалидированных результатов.


- В payload включаются:

  - Названия команд/игроков

  - Дата (в московском времени)

  - Тотал (если удалось вычислить)

  - H2H (личные встречи)

  - Сырые снитпеты и fallback-заметки


- Fallback между LLM реализован: если основной не отвечает — пробуем следующий по приоритету.


- **Футбол**: голы, тотал, H2H, составы, травмы, дисквалификации, xG, pressing, тактика, серия побед/поражений

- **Хоккей**: шайбы, тотал, H2H, составы, травмы, спецбригады, серия побед/поражений

- **Баскетбол**: очки, тотал, H2H, составы, травмы, темп, эффективность, подборы

- **Волейбол**: сеты, тотал, H2H, составы, глубина состава, атака, приём

- **Теннис**: сеты, тотал, H2H, покрытие, физика, психология, серия побед

- **Настольный теннис**: сеты, тотал, H2H, стиль, выносливость

- **ММА/Бокс**: метод победы, раунд, антропометрия, серия побед, стиль боя

- **Киберспорт (CS2, Dota2, LoL, Valorant)**: карты, тотал, H2H, map pool, драфт, мета, форма лидеров, замены, стратегия


## 7.5. Используемые библиотеки и API

- **aiogram** — Telegram-бот

- **google-genai** — Google Gemini API

- **groq** — Groq API

- **openai** — SambaNova/DeepSeek (через совместимый API)

- **hltv** — киберспорт (CS2)

- **sportsipy** — футбол, хоккей, баскетбол (официальная статистика)

- **cloudscraper, requests, beautifulsoup4, lxml** — парсинг сайтов

- **ddgs, duckduckgo_search** — DuckDuckGo поиск

- **tavily-python, exa-py** — Exa/Tavily поиск

- **python-dotenv** — переменные окружения

- **sqlite3** — хранение пользовательских данных


## 7.6. Вывод пользователю

- Ответ всегда содержит:

  - Победителя

  - Вероятность

  - Прогноз на тотал

  - Итоговый счёт

  - Рекомендованный % ставки

  - Краткий анализ по каждой стороне

  - Ключевые факторы и H2H


- Все даты приводятся к московскому времени

- Формат ответа стандартизирован и одинаков для всех дисциплин
