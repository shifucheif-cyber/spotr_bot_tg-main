# Архитектура Telegram-бота спортивной аналитики

## Общая логика работы

Бот принимает запросы пользователей в Telegram, собирает данные о спортивных событиях (дисциплина, команды/игроки, дата), агрегирует информацию из внешних источников, формирует аналитический прогноз с помощью LLM (Google Gemini через Google AI Studio) и возвращает структурированный ответ с вероятностями, счётом и рекомендациями.

## Используемый стек

- Python 3.10+
- aiogram (асинхронный Telegram Bot API)
- httpx (асинхронные HTTP-запросы)
- Google AI Studio (интеграция с Gemini через API)
- SQLite (хранение пользовательских данных)
- Markdown (документация и промпты)
- Внешние спортивные API и парсеры

## Структура файлов и назначение модулей

- `bot.py`
  Точка входа. Инициализация бота, запуск диспетчера, регистрация хендлеров.

- `data_router.py`
  Маршрутизация и обработка пользовательских запросов, определение дисциплины, команд и даты.

- `services/`
  Каталог с сервисами для каждой дисциплины и вспомогательными модулями.

  - `basketball_service.py`, `football_service.py`, `hockey_service.py`, ...
    Логика сбора и агрегации данных по конкретной дисциплине.

  - `data_fetcher.py`
    Асинхронный сбор данных из внешних API и парсеров.

  - `search_engine.py`
    Универсальный слой поиска и агрегации информации по событиям.

  - `betting_calculator.py`
    Расчёт вероятностей, формирование рекомендаций по ставкам.

  - `response_formatter.py`
    Форматирование финального ответа для Telegram.

  - `external_source.py`
    Работа с внешними источниками данных (API, сайты).

  - `logging_utils.py`
    Логирование событий и ошибок.

  - `user_store.py`
    Работа с базой данных пользователей (SQLite).

  - `prompts.py`
    Загрузка и подготовка промптов для LLM.

  - `e2e_summary.py`
    Сбор и анализ итоговой статистики.

  - `name_normalizer.py`
    Приведение названий команд и игроков к единому формату.

  - `match_finder.py`
    Поиск и сопоставление событий по данным пользователя.

- `prompts/`
  Markdown-файлы с промптами для разных дисциплин и общий контракт ответа.

- `requirements.txt`
  Список зависимостей Python.

- `runtime.txt`
  Версия Python для деплоя.

- `bot_data.sqlite3`
  Локальная база данных пользователей и истории запросов.

- `README.md`
  Краткая инструкция по запуску и использованию.

## Процесс обработки запроса

1. Пользователь отправляет запрос в Telegram (например, "Футбол, Реал — Барселона, 10 апреля").
1. `bot.py` принимает сообщение и передаёт его в `data_router.py`.
1. `data_router.py` определяет дисциплину, команды и дату, затем выбирает нужный сервис из каталога `services/`.
1. Сервис (например, `football_service.py`) инициирует сбор данных через `data_fetcher.py` и `search_engine.py`.
1. `data_fetcher.py` асинхронно обращается к внешним API и парсит сайты для получения актуальной информации (результаты, составы, травмы, H2H и т.д.).
1. Собранные данные нормализуются и агрегируются.
1. Сформированный датасет и промпт передаются в Google AI Studio (Gemini) через API.
1. LLM возвращает структурированный прогноз (JSON + текстовый анализ).
1. `betting_calculator.py` и `response_formatter.py` проверяют корректность, рассчитывают вероятности, форматируют финальный ответ.
1. Ответ отправляется пользователю в Telegram.

```text
📊 Матч: Реал Мадрид vs Барселона

📋 Факты из данных:
- (Реал не проигрывает 7 матчей → преимущество Реал)
- (Барселона без двух ключевых защитников → преимущество Реал)
- (Последние 3 очные встречи — ничьи → нейтрально)

📝 Анализ:
• Реал: стабильная форма, сильный состав
• Барселона: проблемы в обороне

📈 Вероятность победы (1-я сторона): 62%
🏆 Победитель: Реал Мадрид
🔢 Прогноз по счету: 2:1
💰 Рекомендуемый % от банка: 3%

"win_probability_team2": 38,
"draw_probability": 0,
"recommended_bet_size": 3,
"confidence_score": 0.85,

"exact_score": "2:1",

```

## Принципы и стандарты

- Весь код асинхронный (async/await, httpx).
- Строгая обработка ошибок (429, 404, таймауты).
- Все внешние запросы — только через httpx.
- Вся логика поиска и агрегации вынесена в сервисы.
- Прогнозы и формат ответов стандартизированы (см. prompts/common_suffix.md).
- Документация и промпты соответствуют markdownlint.
