# PROJECT AGENT AUDIT

Дата: 2026-04-12
Проект: Telegram-бот аналитики спортивных событий
Формат: подробный инженерный аудит рабочего репозитория
Среда разработки: Windows 11
Целевая среда deployment: Ubuntu 22.04
Статус репозитория на момент аудита: runtime переведён на PostgreSQL, тестовый прогон подтверждён, выявлены критичные concurrency и deployment-риски

## 1. Executive Summary

Этот проект представляет собой Telegram-бота на aiogram 3.x, который принимает запрос пользователя на анализ спортивного события, определяет дисциплину, собирает внешние данные, формирует агрегированный контекст, передаёт его в LLM, извлекает вероятностные и betting-поля из ответа модели и возвращает пользователю форматированный прогноз.

На момент аудита проект находится в рабочем состоянии как development/runtime-система и уже прошёл подтверждённый тестовый прогон на PostgreSQL-only архитектуре. Подтверждённый факт: полный тестовый набор собран и ранее прошёл без падений: 251 tests collected, 251 passed. Это сильный сигнал, что базовая архитектура, большинство контрактов модулей и основная функциональность согласованы между собой.

При этом тестовая зелёная сборка не означает production-ready состояние. Основные риски находятся не в синтаксисе и не в очевидных импортных ошибках, а в областях конкурентного доступа, атомарности операций, отказоустойчивости старта и надёжности deployment-path для Ubuntu 22.04. В частности, storage-логика вокруг суточных лимитов и промокодов имеет race-condition класс, а orchestration-слой в bot.py содержит уязвимость на уровне управления пользовательскими семафорами. Дополнительно setup_server.sh требует аккуратной доработки, потому что shell-эскейпинг и запись DATABASE_URL остаются потенциальной точкой поломки на реальном Linux-хосте.

Итоговая инженерная оценка следующая:

- Проект работоспособен как текущая система и проходит тесты.
- Архитектура в целом понятна, модульно разделена и уже пригодна для дальнейшей эволюции.
- Основная бизнес-логика и математическая логика непротиворечивы, но вероятностная часть опирается на LLM и не является статистически калиброванной моделью.
- До production deployment на Ubuntu 22.04 желательно устранить как минимум 4 блока риска: atomic daily limits, promo redemption locking, semaphore lifecycle in bot.py, safe deployment/bootstrap scripting.

## 2. Scope And Method

Аудит выполнен по рабочему репозиторию как по исполняемому приложению, а не только по документации. Анализ охватывает:

- основной runtime-поток
- модули маршрутизации и orchestration
- storage-слой PostgreSQL
- search/retrieval pipeline
- кэширование и фазовую модель событий
- математическую логику betting recommendations
- форматирование ответа
- bootstrap LLM и preflight
- test harness и подтверждение поведения
- deployment-путь под Ubuntu 22.04

При формировании выводов использовались ранее подтверждённые факты:

- runtime переведён на PostgreSQL-only архитектуру
- следы SQLite из рабочей логики удалены
- тестовая инфраструктура адаптирована под PostgreSQL
- полный прогон тестов завершался успешно

Критичные выводы в этом отчёте опираются не на предположения, а на проверенные участки логики в ключевых файлах проекта.

## 3. Repository-Level Architecture

Высокоуровневый runtime-поток проекта:

1. Пользователь пишет запрос в Telegram.
2. bot.py принимает обновление и ведёт пользователя через FSM/state flow.
3. data_router.py определяет дисциплину и соответствующий сервис анализа.
4. sport service и data_fetcher.py собирают контекст по матчу.
5. search_engine.py обращается к поисковым источникам и валидирует материалы.
6. llm_clients.py даёт доступ к LLM-провайдерам; bot.py организует fallback/rotation.
7. betting_calculator.py извлекает вероятность, коэффициенты, score и вычисляет betting summary.
8. response_formatter.py собирает безопасный итоговый Telegram-ответ.
9. user_store.py фиксирует ограничения, подписки, промокоды и платёжные состояния.

Архитектурно проект разделён достаточно разумно:

- bot.py: orchestration и Telegram runtime
- data_router.py: тонкая маршрутизация по дисциплинам
- services/*_service.py: domain-specific слой по видам спорта
- services/data_fetcher.py: единая orchestration-точка сбора данных
- services/search_engine.py: retrieval pipeline
- services/analysis_cache.py, services/event_phase.py: кэш и модель времени жизни данных
- services/betting_calculator.py: извлечение и расчёты
- services/response_formatter.py: контракт ответа
- services/user_store.py: persistence и business restrictions

Это хорошая база: логика не расползлась по одному файлу, а runtime и вычисления разделены. Основные проблемы находятся не в декомпозиции, а в correctness деталей.

## 4. Key Module Analysis

### 4.1 bot.py

Роль модуля:

- точка входа приложения
- инициализация aiogram runtime
- обработка пользовательских запросов
- координация пайплайна анализа
- запуск preflight и периодических background-задач
- выбор и переключение LLM-провайдеров

Сильные стороны:

- логика orchestration централизована
- асинхронная работа согласована с остальным стеком
- интеграция кэшей переведена на async API
- startup и runtime читаются как единый рабочий контур

Слабые стороны:

- lifecycle управления пользовательскими семафорами недостаточно надёжен
- есть риск гонки при создании/удалении semaphore-объектов на пользователя
- устойчивость к частичным ошибкам в асинхронном пайплайне требует дополнительной жёсткости

Инженерный вывод:

bot.py является функциональным, но именно здесь сосредоточены orchestration-risks. Это не проблема архитектуры как таковой, а проблема качества конкурентного управления состоянием во время одновременных пользовательских запросов.

### 4.2 data_router.py

Роль модуля:

- классификация входного текста по дисциплинам
- выбор соответствующего сервиса

Оценка:

- модуль тонкий и по форме корректный
- важна правильная упорядоченность маршрутов, чтобы специфичные виды спорта не перекрывались более общими
- логика маршрутизации не выглядит критичным источником ошибок при текущем масштабе

Вывод:

data_router.py выполняет свою роль и не является основным источником рисков. Это скорее стабильный glue layer.

### 4.3 services/user_store.py

Роль модуля:

- PostgreSQL persistence layer
- хранение пользователей, лимитов, подписок, промокодов, платёжных состояний
- инициализация схемы и миграционная логика

Сильные стороны:

- runtime-конфигурация приведена к PostgreSQL-only модели
- asyncpg-подход соответствует остальному async стеку
- модуль концентрирует всю stateful бизнес-логику в одном месте

Критичные проблемы:

- проверка суточного лимита и увеличение счётчика не образуют одну атомарную операцию
- активация промокода уязвима к конкурентному повторному использованию при неблагоприятном времени запросов
- логика correctness зависит от порядка вызовов приложения, а не только от БД-гарантий

Вывод:

Это самый критичный модуль проекта с точки зрения production correctness. Он не выглядит сломанным при одиночных сценариях, но при реальной многопоточности/многопользовательской нагрузке способен давать неверный результат без явного исключения.

### 4.4 services/search_engine.py

Роль модуля:

- сбор источников
- валидация/фильтрация результатов поиска
- orchestration поисковых провайдеров

Сильные стороны:

- retrieval pipeline уже отделён от Telegram runtime
- логика расширения провайдеров масштабируема
- предусмотрены fallback-источники

Риски:

- асинхронная обработка множества результатов чувствительна к частичным ошибкам и исключениям внутри gather/pipeline
- корректность финального набора источников зависит от того, как обрабатываются нештатные ответы внешних API

Вывод:

Модуль архитектурно правильный, но требует повышенного внимания к exception isolation и стабильности degradation path, особенно когда один внешний поставщик отвечает ошибкой, медленно или нестабильно.

### 4.5 services/data_fetcher.py

Роль модуля:

- orchestration сбора данных о событии
- фазово-зависимое кэширование

Сильные стороны:

Вывод:

Модуль выглядит содержательно зрелым. Его работа логически согласована с фазовой моделью проекта. Основной риск здесь не в самой идее, а в корректности входных данных и в зависимостях от search pipeline.

Оценка:

- одна из наиболее чистых и логически стройных частей проекта
- даёт понятный bridge между временем матча и кэш-политикой
- уменьшает количество магических TTL-решений в остальных частях системы

Вывод:

Фазовая модель удачна. Она повышает объяснимость системы и делает кеширование более инженерно обоснованным.

### 4.7 services/betting_calculator.py

Роль модуля:

- извлечение вероятностей, коэффициентов и прогнозных данных из текста модели

Сильные стороны:

- если LLM даёт формально корректное, но плохо обоснованное число, downstream-математика будет консистентной, но не обязательно истинной

Вывод:

Математика внутри модуля в целом согласована. Главная слабость не в формулах, а в происхождении исходной вероятности.

## 5. Confirmed Findings By Severity

### 5.1 Critical: non-atomic daily limit control

Где:

- services/user_store.py

Суть проблемы:

Проверка возможности сделать запрос и инкремент счётчика запросов разделены логически и могут выполняться раздельно. Если два запроса одного пользователя проходят почти одновременно, оба могут увидеть, что лимит ещё не исчерпан, и затем оба увеличить счётчик. Это нарушает бизнес-ограничение.

Почему это реально критично:

- ошибка не обязана проявляться в тестах одиночного потока
- при concurrency это приводит к ложному разрешению лишних запросов
- проблема затрагивает платные ограничения и fairness бизнес-модели

Влияние:

- перерасход лимита
- некорректный контроль подписки и бесплатного доступа
- потенциальные финансовые и репутационные потери

Что исправлять:

- свести check+increment к одной атомарной операции в БД
- использовать транзакцию с корректной блокировкой или UPSERT/UPDATE с условием и проверкой affected rows

### 5.2 Critical: promo activation race condition

Где:

- services/user_store.py

Суть проблемы:

Активация промокода не гарантирует монопольную фиксацию использования в условиях конкурентного доступа. Один и тот же код при неудачном timing может быть принят более одного раза до полного отражения изменения состояния.

Почему это критично:

- напрямую затрагивает денежную/подписочную бизнес-логику
- может использоваться повторно при гонке запросов

Влияние:

- повторное погашение промокодов
- некорректное начисление прав доступа
- расхождение фактического и ожидаемого состояния в БД

Что исправлять:

- row-level locking
- atomic update с условием неиспользованности промокода
- строгая проверка результата обновления внутри транзакции

### 5.3 High: semaphore lifecycle race in bot runtime

Где:

- bot.py

Суть проблемы:

Управление пользовательскими семафорами недостаточно защищено от гонок при параллельных запросах и cleanup-сценариях. Это способно приводить к ситуации, когда guard на пользователя ведёт себя нестабильно: дублируется, удаляется раньше времени или не обеспечивает ожидаемую сериализацию.

Почему это важно:

- это защита от конкурирующих анализов одного пользователя
- ошибка в этом слое может усиливать проблемы storage-уровня

Влияние:

- неконсистентный runtime flow
- потенциальное двойное выполнение дорогих операций
- усиление race condition в user_store.py

Что исправлять:

- сделать жизненный цикл per-user semaphore строго детерминированным
- исключить незащищённое создание и cleanup shared runtime state
- централизовать acquire/release/cleanup policy

### 5.4 High: deployment script fragility on Ubuntu 22.04

Где:

- setup_server.sh

Суть проблемы:

Shell-логика записи env и модификации конфигурации может сломаться на специальных символах, кавычках, URL-encoding и отличиях окружения. DATABASE_URL и смежные переменные особенно чувствительны.

Почему это важно:

- это production bootstrap path
- локально проект может быть рабочим, но deployment рухнет на этапе автоматизации

Влияние:

- некорректный .env
- невозможность старта после деплоя
- трудно диагностируемые ошибки в Linux среде

Что исправлять:

- убрать хрупкие sed-шаблоны там, где возможны символы URL/пароля
- использовать безопасную запись env-файла
- проверять итоговый .env на корректность после генерации

### 5.5 Medium: partial failure handling in retrieval pipeline

Где:

- services/search_engine.py

Суть проблемы:

Асинхронный сбор результатов из нескольких внешних источников требует особенно осторожной обработки частичных ошибок. Если исключения внутри gather/validation flow нормализуются непоследовательно, то итоговая деградация может быть либо слишком жёсткой, либо слишком тихой.

Влияние:

- частичная потеря качества анализа
- нестабильное поведение при проблемах провайдера
- сложность диагностики degraded mode

Что исправлять:

- выровнять policy обработки исключений
- логировать деградации на уровне источника
- отделять provider failure от total pipeline failure

### 5.6 Medium: startup readiness transparency

Где:

- services/llm_clients.py
- preflight_check.py
- bot.py

Суть проблемы:

Даже если диагностика инициализации провайдеров есть, система должна предельно ясно различать состояния "полностью готова", "готова частично" и "не готова". В production это критично для оператора.

Влияние:

- неоднозначный startup state
- риск старта без реально доступных провайдеров

Что исправлять:

- fail-fast при полном отсутствии работоспособных клиентов
- чёткий startup report
- единая policy критичности preflight-проверок

## 6. Mathematical Logic Audit

### 6.1 Phase Model

В проекте присутствует важная инженерная идея: жизненный цикл события моделируется через отдельные фазы, а не через один общий TTL. Это даёт системе несколько преимуществ:

- до матча и в ранней фазе можно хранить данные умеренно долго
- в live/near-live зонах TTL должен быть меньше
- после завершения события результаты можно ещё некоторое время возвращать из кэша
- полностью устаревшие события можно отбрасывать как EXPIRED

Это корректная и зрелая постановка задачи. Она делает кэширование привязанным к природе спортивного события, а не к случайной технической константе.

### 6.2 Probability Extraction

betting_calculator.py преобразует LLM-output в числовые поля. Сам подход технически понятен: сначала попытка извлечь структурированные или полу-структурированные данные, затем нормализация диапазона и расчёт downstream-метрик.

Проблема здесь не в формулах, а в источнике исходной вероятности. Если probability приходит из языковой модели, она не является вероятностью в строгом статистическом смысле, если отдельно не проводилась калибровка. Поэтому математическая часть проекта формально непротиворечива, но epistemically limited.

Практический смысл этого вывода:

- fair odds можно считать корректно относительно заданной вероятности
- edge можно считать корректно относительно fair odds и market odds
- fractional Kelly можно считать корректно относительно ожидаемого edge
- но качество этих чисел зависит от качества исходной вероятности, а не только от формулы

### 6.3 Fair Odds And Value Logic

Расчётная логика выглядит адекватной для advisory-style системы:

- есть извлечение вероятности
- есть перевод в fair odds
- есть сравнение с внешним коэффициентом
- есть рекомендационная интерпретация edge/value

При этом нужно прямо понимать ограничение: это не quantitative trading engine и не verified bookmaker model. Это decision-support поверх LLM + retrieved context.

### 6.4 Kelly-Like Stake Sizing

Использование fractional Kelly или Kelly-like логики в таком контексте допустимо как эвристика, если соблюдены две оговорки:

- stake recommendation должна быть консервативной
- пользователю нельзя неявно внушать, что underlying probability statistically validated

Если в проекте уже есть клэмпы и ограничения размера ставки, это инженерно правильный выбор.

Итог по математике:

- внутренняя арифметика выглядит связной
- главная граница качества проходит не по формулам, а по качеству исходной вероятности от LLM

## 8. Windows vs Ubuntu 22.04 Audit View

### 8.1 Что относится только к Windows/dev environment

- специфические проблемы pytest-postgresql на Windows
- необходимость override startup command для локального тестового кластера PostgreSQL
- отсутствие os.killpg в Windows и соответствующий test shim

Это не production-баги самого приложения. Это инфраструктурные ограничения локальной среды тестирования.

### 8.2 Что реально важно для Ubuntu deployment

- корректная генерация DATABASE_URL и env
- надёжность setup_server.sh
- наличие нужной версии Python
- предсказуемый startup policy
- fail-fast при нехватке критичных env и провайдеров
- доступность PostgreSQL и корректный bootstrap схемы

### 8.3 Что cross-platform и опасно в любой среде

- race conditions в user_store.py
- race condition в bot.py semaphore lifecycle
- неидеальная прозрачность частичных отказов retrieval/LLM pipeline

Итоговый смысл:

Windows-специфичные проблемы были в основном тестовой инфраструктурой. Настоящие архитектурные риски платформенно независимы и проявятся и на Linux.

## 11. Priority Remediation Order

Если переходить от аудита к исправлениям, правильный порядок такой:

1. Сделать atomic daily limit operation в services/user_store.py.
2. Сделать atomic promo redemption с row-level locking в services/user_store.py.
3. Перестроить lifecycle per-user semaphore в bot.py.
4. Усилить setup_server.sh и безопасную генерацию env.
5. Выровнять startup/preflight policy для случая отсутствия рабочих LLM providers.
6. Добавить целевые concurrency tests для user_store.py и bot runtime.
7. Добавить controlled failure tests для retrieval pipeline.

Именно такой порядок закрывает сначала correctness и business-risk, потом deployment-hardening, потом observability и устойчивость.

## 12. Final Audit Conclusion

Финальный вывод по проекту точный и прямой:

Кодовая база в целом рабочая, логически связная и уже прошла важный этап оздоровления после перехода к PostgreSQL-only архитектуре. Архитектура проекта разумна: orchestration, storage, retrieval, formatting и domain services в основном разделены правильно. Тестовая база подтверждает, что значительная часть системы работает согласованно.

Но это не тот случай, когда "всё зелёное в тестах" означает "можно без опасений выкатывать в production". В проекте подтверждены критичные инженерные риски именно в тех зонах, которые редко ловятся обычными тестами: атомарность ограничений доступа, конкурентная активация промокодов, lifecycle shared runtime state и надёжность deployment bootstrap.

Поэтому итоговая оценка следующая:

- как рабочий проект и основа для дальнейшей разработки система состоятельна
- как production deployment target для Ubuntu 22.04 система близка, но ещё не hardened
- ближайший путь к production-ready состоянию не требует переписывания архитектуры, а требует точечных исправлений в correctness-critical узлах

Это хороший признак. Корень проблем здесь не в плохой архитектуре, а в нескольких конкретных инженерных недожатостях, которые можно устранить без слома всей системы.
