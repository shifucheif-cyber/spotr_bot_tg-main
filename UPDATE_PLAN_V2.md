# UPDATE PLAN V2 — Roadmap будущих улучшений

## 1. ✅ asyncpg для user_store.py

- ✅ Заменили `psycopg2-binary` (sync) на `asyncpg` (полный неблокирующий PostgreSQL-драйвер)
- ✅ Все функции `user_store.py` — `async def`, PostgreSQL через `asyncpg.create_pool()`, SQLite через `asyncio.to_thread()`
- ✅ `ThreadPoolExecutor` удалён из `bot.py`
- ✅ Все вызовы user_store в bot.py обёрнуты в `await`
- ✅ Тесты переведены на `IsolatedAsyncioTestCase`

## 2. Celery + Redis: очереди для тяжёлых запросов

- При пиковой нагрузке — ставить анализы в очередь
- Пользователь получает: «⏳ Вы 15-й в очереди, примерное время: 2 мин»
- Celery worker обрабатывает задачи, результат отправляется по готовности
- Redis как брокер сообщений и кэш

## 3. ✅ Разделение search_engine.py на search_providers/

- ✅ Вынесено в `services/search_providers/`:

  - `helpers.py` — утилиты нормализации/валидации
  - `providers.py` — функции провайдеров (DDG, Serper, Exa, Tavily)
- ✅ `search_engine.py` — тонкий оркестровщик + реэкспорт имён (обратная совместимость mock-путей)
- ✅ Все 159 тестов проходят без изменений

## 5. ✅ Round-robin LLM балансировка

- Будущее улучшение: мониторинг latency и автоматический выбор самого быстрого

## 6. Кэширование результатов поиска

## 📅 Задачи из аудита (изменить сегодня или завтра)

### 7. Telegram-Таймауты и Блокировки (Критично для Production)
- **Риск:** Сейчас сбор данных + анализ LLM может занимать 10-30 секунд. Telegram отваливается или дублирует webhook/polling запросы, если бот не отвечает слишком долго.
- **Решение:** Разделение на Worker'ы (Celery/Redis) из плана V2 — это абсолютный маст-хэв перед запуском на большую аудиторию. Бот должен моментально отвечать «Анализ запущен...» и отдавать тяжелую задачу воркеру.

### 8. Отсутствие кэширования (Средний риск)
- **Риск:** Если 50 человек за час спросят "Реал vs Барселона", бот 50 раз спарсит интернет и 50 раз обратится к LLM. Это сожгет API-лимиты и доллары.
- **Решение:** Обычный Redis-cache с TTL около 2-4 часов (вполне достаточно для прематч-статистики). Ключ: `f"analysis:{discipline}_{match_hash}"`. Опять же, это упомянуто в п.6 твоего плана V2.

### 9. Отсутствие миграций БД (Низкий риск)
- **Контекст:** Сейчас таблицы создаются через `CREATE TABLE IF NOT EXISTS` в коде (`init_user_store`).
- **Для будущего:** Если понадобится сложно менять типы колонок удалять старые поля, этот подход станет опасным. Стоит присмотреться к Alembic (или легковесным аналогам типа yoyo-migrations), когда проект обрастет сложной бизнес-логикой и платной подпиской.

### 10. Логирование и Мониторинг (Средний risk)
- **Контекст:** `logger = logging.getLogger(__name__)` настроен хорошо, но при переходе на сервер тебе будет сложно грепать text-файлы.
- **Для будущего:** Прикрутить интеграцию с Sentry (для отлова Exception'ов) и, возможно, писать структурированные JSON-логи.
